## 5. Consumer（消费者）— 你项目中 Spark 充当了 Consumer

你的项目比较特殊：消费者不是 Spring Kafka Consumer，而是 Spark Structured Streaming。但原理一样，我两种都教你。

### 5.1 Spring Kafka Consumer 的标准写法

```java
@Component
public class RecommendationConsumer {

      // 声明式消费 — 最简单的方式
      @KafkaListener(
          topics = "${kafka.topic.recommendations}",
          groupId = "recommendation-result-group",
          concurrency = "3"  // 3个线程并行消费
      )
      public void consumeRecommendation(
              @Payload String message,
              @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
              @Header(KafkaHeaders.OFFSET) long offset) {

          RecommendationResult result = objectMapper.readValue(message,
              RecommendationResult.class);

          // 业务处理：把推荐结果写入Redis/数据库供前端查询
          recommendationService.saveResult(result);

          log.info("消费推荐结果: userId={}, partition={}, offset={}",
              result.getUserId(), partition, offset);
      }

}

```

### 5.2 核心参数解释

```java
@KafkaListener(
    topics = "recommendations", // 订阅哪个Topic
    groupId = "result-group",   // 消费组名（见下节）
    concurrency = "3"           // 并发消费线程数
    // concurrency不能超过分区数，超了也只有分区数个线程能工作
)

```

### 5.3 手动 Offset 提交（企业必备）

```java
@KafkaListener(topics = "user-events", groupId = "analytics-group")
public void consume(ConsumerRecord<String, String> record,
                    Acknowledgment ack) {
    try {
        processEvent(record.value());
        ack.acknowledge(); // 处理成功才提交offset
    } catch (Exception e) {
        // 处理失败，不提交offset → 下次重新消费这条消息
        log.error("处理失败，将重试", e);
        // 可以加入死信队列逻辑
    }
}

```

> **为什么不用自动提交？** > 自动提交是"拉取消息后就提交"，如果你处理到一半应用挂了，消息已经标记为已消费，重启后就跳过了 → 数据丢失。手动提交保证"处理完了才提交"。

## 6. Consumer Group（消费组）— 并行消费的核心机制

### 你的配置

```properties
spring.kafka.consumer.group-id=recommendation-consumer-group

```

**消费组的核心规则只有一条：**
**一个分区在同一消费组内只能被一个消费者消费**

看图理解：

```text
Topic: user-events (6个分区)

消费组A: analytics-group (3个消费者实例)
┌────────────┐
│ Consumer-1 │ ← Partition 0, 1 (每个消费者分到2个分区)
│ Consumer-2 │ ← Partition 2, 3
│ Consumer-3 │ ← Partition 4, 5
└────────────┘

消费组B: spark-streaming-group (6个消费者实例)
┌────────────┐
│ Consumer-1 │ ← Partition 0 (每个消费者分到1个分区)
│ Consumer-2 │ ← Partition 1
│ ...        │
│ Consumer-6 │ ← Partition 5
└────────────┘

```

**关键理解：**

- 消费组A和消费组B互不影响，各自独立消费全量数据
- 组A有3个消费者、6个分区 → 每个消费者分到2个分区
- 如果组A加第4个消费者 → 重新分配，每人1-2个分区
- 如果组A加到第7个消费者 → 有1个空闲，因为只有6个分区

### 你的推荐系统里应该有哪些消费组？

- **消费组1: `spark-streaming-group**`
- ↑ Spark Structured Streaming 消费 `user-events`
- ↑ 做实时行为分析和推荐计算

- **消费组2: `analytics-group**`
- ↑ 分析服务消费 `user-events`
- ↑ 做离线统计、报表、数据仓库写入

- **消费组3: `recommendation-result-group**`
- ↑ 后端服务消费 `recommendations`
- ↑ 把Spark产出的推荐结果写入Redis供前端查询

同一份数据被不同消费组消费多次——这就是 Kafka 比传统消息队列强大的地方。RabbitMQ 里消息被消费就没了，Kafka 的消息存在磁盘上，不同消费组各自维护自己的消费进度。
