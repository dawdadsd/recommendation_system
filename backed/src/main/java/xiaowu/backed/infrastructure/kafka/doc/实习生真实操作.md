## 实习生真实接触的 Kafka 场景

**现实：实习生在公司里干什么**

- **核心链路**（订单、支付、推荐排序） → 正式员工P6/P7负责，实习生别想碰
- **数据链路**（日志、埋点、监控） → 实习生最常接触的
- **内部工具**（管理后台、数据平台） → 实习生第二常做的
- **边缘业务**（活动页、运营工具） → 实习生练手的

---

### 场景1：埋点日志采集管道（最最最常见）

**你的leader给你的任务：**

> "APP端的用户行为埋点现在直接写ES，查询太慢了。你把中间加一层Kafka缓冲，写一个消费者把数据落到Hive里，数据分析团队要用。"

**你实际写的代码：**

```text
APP → 埋点SDK → 后端埋点收集接口 → Kafka → 你写的Consumer → Hive/ClickHouse
                                   ↑
                             已有的，你不用管

你的工作范围：Kafka → Hive 这一段

```

```java
// 你写的消费者，从Kafka读埋点数据，攒一批写入Hive
@Component
public class TrackEventConsumer {

      private final List<TrackEvent> buffer = new ArrayList<>();
      private static final int BATCH_SIZE = 1000;

      @KafkaListener(topics = "user-track-events", groupId = "hive-sink-group")
      public void consume(ConsumerRecord<String, String> record,
                          Acknowledgment ack) {
          TrackEvent event = JSON.parseObject(record.value(), TrackEvent.class);
          buffer.add(event);

          if (buffer.size() >= BATCH_SIZE) {
              hiveWriter.batchInsert(buffer);  // 攒够1000条写一次
              buffer.clear();
              ack.acknowledge();
          }
      }

}

```

**你会遇到的真实问题：**

- **问题：** buffer里攒了800条还没到1000条，这时候应用重启了，800条丢了
- **你的leader会教你：**
- → 加一个定时刷新机制，每30秒不管攒了多少条都写一次
- → 或者用Kafka Connect（不用自己写Consumer了，配置化就能把数据导到Hive）

这就是你面试时能讲的"踩坑经历"

### 场景2：操作日志审计

**你的leader给你的任务：**

> "运营后台现在没有操作日志，产品说要能查到谁在什么时间改了什么配置。你加一下。"

```text
运营人员在后台改配置 → 你的AOP拦截器 → 发Kafka → 你的Consumer → 写MySQL审计表

```

```java
// 你写的AOP切面，拦截所有运营操作，发到Kafka
@Aspect
@Component
public class AuditLogAspect {

      @Autowired
      private KafkaTemplate<String, String> kafkaTemplate;

      @Around("@annotation(auditLog)")
      public Object audit(ProceedingJoinPoint joinPoint, AuditLog auditLog)
              throws Throwable {
          // 记录操作前的状态
          String operator = SecurityContext.getCurrentUser();
          String action = auditLog.value();
          String params = JSON.toJSONString(joinPoint.getArgs());

          Object result = joinPoint.proceed();  // 执行原方法

          // 操作成功后发送审计日志到Kafka
          AuditEvent event = new AuditEvent(operator, action, params,
                                            LocalDateTime.now());
          kafkaTemplate.send("audit-log", operator, JSON.toJSONString(event));

          return result;
      }

}

// 使用方式：运营接口上加个注解就行
@AuditLog("修改商品价格")
@PostMapping("/product/updatePrice")
public Result updatePrice(@RequestBody PriceUpdateDTO dto) { ... }

```

**你会遇到的真实问题：**

- **问题：** Kafka发送失败了（比如Kafka集群短暂不可用），但原方法已经执行成功了，审计日志丢了
- **你的思考过程：**
- → 方案1：发送失败就回滚原操作？不行，审计日志不应该影响主业务
- → 方案2：发送失败写本地文件兜底，后面补偿重发
- → 方案3：先写本地数据库，再异步同步到Kafka

- **你的leader会告诉你：** 用方案2，这叫"旁路日志不影响主链路"原则

### 场景3：异步通知解耦

**你的leader给你的任务：**

> "用户注册成功后要发欢迎短信、发优惠券、初始化用户画像。现在全写在注册接口里，注册一次要3秒。你用Kafka解耦一下。"

**改造前（同步，3秒）：**
注册接口 → 写DB → 发短信(800ms) → 发优惠券(600ms) → 初始化画像(500ms) → 返回

**改造后（异步，200ms）：**

```text
注册接口 → 写DB → 发Kafka消息 → 返回（200ms搞定）
                   ↓
                   Consumer1 → 发短信
                   Consumer2 → 发优惠券
                   Consumer3 → 初始化画像

```

```java
// 你改造注册接口：只写库+发消息
@PostMapping("/register")
public Result register(@RequestBody RegisterDTO dto) {
      User user = userService.createUser(dto); // 写DB

      // 发一条注册事件到Kafka，后续处理全部异步
      UserRegisteredEvent event = new UserRegisteredEvent(
          user.getId(), user.getPhone(), LocalDateTime.now());
      kafkaTemplate.send("user-registered",
                         String.valueOf(user.getId()),
                         JSON.toJSONString(event));

      return Result.success(user.getId());

}

// 短信消费者（可能是另一个同事写的，也可能是你写的）
@KafkaListener(topics = "user-registered", groupId = "sms-group")
public void sendWelcomeSms(String message) {
      UserRegisteredEvent event = JSON.parseObject(message,
      UserRegisteredEvent.class);
      smsService.send(event.getPhone(), "欢迎注册xxx");
}

// 优惠券消费者（另一个消费组，独立消费同一条消息）
@KafkaListener(topics = "user-registered", groupId = "coupon-group")
public void grantCoupon(String message) {
      UserRegisteredEvent event = JSON.parseObject(message,
      UserRegisteredEvent.class);
      couponService.grantNewUserCoupon(event.getUserId());
}

```

**你会遇到的真实问题：**

- **问题：** 短信消费者挂了，用户注册成功了但没收到短信
- **你的排查过程：** 1. 看消费组lag → lag在涨，说明消费者确实没在消费

2. 看消费者日志 → 发现是短信接口限流了，抛异常后消费者线程挂了
3. 修复：加try-catch + 失败重试3次 + 写入死信Topic

```java
// 死信队列（DLQ）模式
@KafkaListener(topics = "user-registered", groupId = "sms-group")
public void sendSms(String message, Acknowledgment ack) {
      try {
          // 正常处理
          smsService.send(...);
          ack.acknowledge();
      } catch (Exception e) {
          retryCount++;
          if (retryCount >= 3) {
              // 重试3次还失败 → 扔进死信Topic，人工处理
              kafkaTemplate.send("user-registered-dlq", message);
              ack.acknowledge(); // 提交offset，不再重试
          }
          // 不acknowledge → 下次重新消费
      }
}

```

### 场景4：数据同步（DB → 缓存/搜索引擎）

**你的leader给你的任务：**

> "商品信息改了之后ES搜索索引没更新，用户搜到的还是旧数据。你监听MySQL变更同步到ES。"

```text
商品表变更 → Canal/Debezium监听binlog → Kafka → 你的Consumer → 更新ES索引
                                          ↑
                                     运维已经搭好了
                   你的工作：写Consumer消费binlog变更消息，更新ES

```

**你会遇到的真实问题：**

- **问题：** 商品改了名字，ES里搜旧名字还能搜到，搜新名字搜不到
- **排查：** Consumer日志显示消费正常，但发现是ES写入后有1秒refresh间隔，不是Kafka的问题

这个排查过程本身就是面试素材——说明你能区分"Kafka的问题"和"下游的问题"

### 总结：实习生的 Kafka 经验长什么样

**面试时这样说：**

> "我在实习期间负责了用户行为埋点的数据管道建设。
> APP端埋点数据通过Kafka投递过来，我写了消费者把数据落到ClickHouse供数据分析团队查询。
> Topic是user-track-events，6个分区，我的消费组有2个实例。
> 遇到过一个问题是消费者攒批写入时，攒到一半应用重启导致数据丢失。后来加了定时刷新机制，每30秒强制flush一次，并且把offset提交改成了手动提交，确保数据写入成功后才提交offset。
> 还有一次消费lag突然飙高，排查发现是ClickHouse写入超时导致消费变慢，临时加了一个消费者实例扩容解决的。"

这段话里有：具体数字、具体问题、排查过程、解决方案。面试官一听就知道你真做过，虽然不是核心链路，但确实在生产环境用过 Kafka。

实习生不需要假装做过支付链路的 Kafka，把埋点管道/日志采集/异步通知这种真实场景讲清楚、讲透，比背10个核心场景的八股文强100倍。
