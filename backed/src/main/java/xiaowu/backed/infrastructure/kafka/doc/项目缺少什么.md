## 适合你项目的3个扩展方向

### 扩展1：写一个 Spring Kafka Consumer — 推荐结果落地

**现状：** Spark 做完行为聚合后，输出到控制台就没了。

**你要做的：** 让 Spark 把推荐结果写回 Kafka 的 `recommendations` Topic，然后你写一个 Spring Consumer 消费这个 Topic，把推荐结果写入 Redis，供前端接口查询。

**完整链路变成：**

```text
模拟器 → Kafka(user-events) → Spark聚合计算 → Kafka(recommendations) → 你的Consumer → Redis
                                                                               ↓
                                                                         前端调接口 → 拿到推荐结果

```

**你会自然遇到的问题：**

- **问题A：Spark写到recommendations Topic的数据格式你没定义清楚**
- → Spark那边输出的是聚合后的JSON，你Consumer这边解析失败
- → **你会学到：** 生产者和消费者之间必须约定Schema

- **问题B：Redis里的推荐结果要怎么组织？**
- → key设计：`recommend:{userId}` → `List<商品ID>`
- → 过期时间：推荐结果时效性短，设30分钟
- → **你会学到：** 消费者的核心工作不是消费消息本身，而是下游的存储设计

- **问题C：Spark可能短时间输出大量聚合结果**
- → 你的Consumer逐条写Redis太慢
- → 改成Pipeline批量写入
- → **你会学到：** 上面讲的"批量写入"模式

> 这就是上面场景里的"ClickHouse入仓"在你项目里的对应版本，只不过下游从ClickHouse换成了Redis。

### 扩展2：订单/评分事件的异步解耦

**现状：** 你有 `product-ratings` Topic，但没有任何代码往里发消息，也没有消费者。

**你要做的：** 模拟一个用户评分场景。用户对商品评分后，发一条消息到 `product-ratings`，多个消费者各自处理。

```text
用户提交评分 → 评分接口 → 写DB → 发Kafka(product-ratings) → 返回成功
                                         ↓
                                消费组A: 更新商品平均分(Redis缓存)
                                消费组B: 喂给Spark更新推荐模型

```

```java
// 评分接口
@PostMapping("/api/ratings")
public Result rateProduct(@RequestBody RatingDTO dto) {
    // 写DB（核心操作，同步）
    ratingMapper.insert(dto);

    // 发Kafka（异步通知下游）
    TransactionSynchronizationManager.registerSynchronization(
        new TransactionSynchronization() {
            @Override
            public void afterCommit() {
                kafkaTemplate.send("product-ratings",
                    String.valueOf(dto.getProductId()),  // key=productId
                    JSON.toJSONString(dto));
            }
        }
    );
    return Result.success();
}

// 消费者：更新商品平均分缓存
@KafkaListener(topics = "product-ratings", groupId = "rating-cache-group")
public void updateRatingCache(String message, Acknowledgment ack) {
    RatingDTO dto = JSON.parseObject(message, RatingDTO.class);

    // 重新计算平均分写入Redis
    Double avg = ratingMapper.selectAvgByProductId(dto.getProductId());
    redisTemplate.opsForValue().set(
        "product:avgRating:" + dto.getProductId(), avg.toString());

    ack.acknowledge();
}

```

**你会自然遇到的问题：**

- **问题A：同一个商品短时间被多人评分，每次都重新算平均分查DB**
- → 短时间内10个人评分 = 10次 `SELECT AVG(rating)`
- → 优化：用Redis `INCR`维护总分和次数，不用每次查DB
- → **你会学到：** Consumer里减少对DB的依赖

- **问题B：product-ratings用compact策略，key怎么设计？**
- → 如果`key=productId`，compact只保留最新一条评分，其他评分全丢
- → 应该`key=userId:productId`，这样每个用户对每个商品保留最新评分
- → **你会学到：** `cleanup.policy=compact`对key设计的要求

> 这就是上面"异步通知解耦"场景在你项目里的真实对应。

### 扩展3：脏数据防御 + 死信队列 + 监控

**现状：** 你的模拟器生成的数据格式完全可控，永远不会出错。但真实场景中数据来源不可信。

**你要做的：** 让模拟器故意产生一定比例的脏数据，然后在 Consumer 端做防御。

```java
// 改造模拟器：5%概率产生脏数据
public BehaviorEventDTO generateEvent() {
    if (random.nextInt(100) < 5) {
        // 模拟脏数据
        return switch (random.nextInt(3)) {
            case 0 -> createEvent(null, 100, "VIEW");      // userId为空
            case 1 -> createEvent(1, -1, "VIEW");          // itemId非法
            case 2 -> createEvent(1, 100, "INVALID_TYPE"); // 行为类型不存在
            default -> createNormalEvent();
        };
    }
    return createNormalEvent();
}

```

**然后你的 Consumer 要做的完整处理：**

```java
@Component
public class RecommendationConsumer {

    private final AtomicLong totalCount = new AtomicLong(0);
    private final AtomicLong failCount = new AtomicLong(0);

    @KafkaListener(topics = "recommendations", groupId = "rec-result-group")
    public void consume(ConsumerRecord<String, String> record,
                        Acknowledgment ack) {
        totalCount.incrementAndGet();

        try {
            RecommendationResult result = parseAndValidate(record.value());

            if (result == null) {
                // 解析失败或校验不通过 → 死信队列
                failCount.incrementAndGet();
                kafkaTemplate.send("recommendations-dlq",
                    record.key(), record.value());
                ack.acknowledge();
                return;
            }

            // 正常处理：写入Redis
            redisPipeline.execute(result);
            ack.acknowledge();

        } catch (Exception e) {
            failCount.incrementAndGet();
            kafkaTemplate.send("recommendations-dlq",
                record.key(), record.value());
            ack.acknowledge();
        }
    }

    // 暴露监控指标
    @Scheduled(fixedRate = 60_000)
    public void reportMetrics() {
        long total = totalCount.get();
        long fail = failCount.get();
        double failRate = total > 0 ? (double) fail / total * 100 : 0;
        log.info("消费统计 | 总计:{} | 失败:{} | 失败率:{:.2f}%",
                 total, fail, failRate);

        if (failRate > 5) {
            log.error("失败率超过5%！请检查上游数据质量");
            // 实际项目里这里发钉钉/飞书告警
        }
    }
}

```
