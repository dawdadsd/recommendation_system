## 一、区分"背八股"和"真用过"的提问策略

背八股的人能告诉你"acks=all保证不丢消息"，真用过的人能告诉你"上次acks=all配了但还是丢了，因为min.insync.replicas没配，broker挂了两台就降级成acks=1了"。

> **核心原则：** 不问"是什么"，问"你遇到了什么"。

### 第一层：开放式场景切入

- ❌ **差的问法：** Kafka如何保证消息不丢失？
- → 背过的人都会答：生产者acks=all，broker副本机制，消费者手动提交offset

- ✅ **好的问法：** 你们生产环境丢过消息吗？怎么发现的？怎么解决的？
- → 没用过的人开始编故事，细节经不起追问
- → 用过的人会说具体的监控告警、排查过程、修复方案

### 第二层：逆向追问细节

候选人说"我负责用户行为事件的Kafka接入"，接着问：

- **Q: 你们Topic的分区数是多少？为什么是这个数字？**
- **背的人：**"12个" （说不出为什么）
- **真的人：**"最开始6个，后来日均事件量从500万涨到5000万，单分区吞吐跟不上，扩到了18个。扩分区那次还踩了坑，因为key哈希变了，同一个userId的消息分布到了不同分区"

- **Q: 扩分区后你们怎么处理key路由变化的？**
- → 这个问题直接过滤90%背八股的人

### 第三层：反常识问题

- **Q: 你们有没有场景是故意用 acks=0 的？**
- **背的人：**"不会，acks=0会丢消息"（教科书答案）
- **真的人：**"有，埋点日志采集那条链路用的acks=0，每秒200万条，丢千分之一可以接受，但换来的是延迟从15ms降到2ms，省了30%的broker资源"

## 二、美团级别的真实任务场景

### 场景1：外卖订单状态变更事件流

**任务背景：**
用户下单 → 商家接单 → 骑手接单 → 取餐 → 送达
每个状态变更都要通知：用户端APP、商家端、骑手端、数据分析、风控系统

**分配给候选人的任务：**
设计订单状态变更的 Kafka 消息链路，要求：

- 日均3000万单，峰值（午高峰）每秒5万条状态变更
- 消息不能丢（用户看不到骑手位置会投诉）
- 同一订单的状态变更必须有序（不能先收到"送达"再收到"取餐"）

**真实会遇到的问题：**

#### 问题1：消费者处理慢，消息积压（Lag飙升）

- **现象：** 午高峰11:30，监控告警 consumer lag 突然从100飙到50万。用户端显示骑手位置延迟3分钟才更新。
- **原因：** 消费者里调了一个下游RPC接口（查骑手位置），那个接口午高峰响应从5ms涨到200ms，消费速度直接降了40倍。
- **解决方案：**

1. **紧急：** 临时扩消费者实例数（从3个扩到12个）
2. **短期：** 消费者里把RPC调用改成异步，先写入本地队列再批量查询
3. **长期：** 把骑手位置查询从消费链路中拆出去，消费者只做"状态写库+推送通知"这种轻量操作，位置信息走独立的Topic和消费链路

#### 问题2：分区扩容导致消息乱序

- **现象：** 原来6个分区，扩到12个后，部分订单出现"骑手已送达"在"骑手已取餐"之前到达。
- **原因：** orderId做key，`hash(orderId) % 6` 和 `hash(orderId) % 12` 分到的分区不一样。扩容瞬间，同一订单的新旧消息落在了不同分区，跨分区无法保证顺序。
- **解决方案：**

1. 停止生产者 → 等消费者消费完所有积压 → 扩分区 → 恢复生产者（简单粗暴，但需要短暂停服）
2. 更好的方案：新建一个12分区的Topic，生产者切到新Topic，旧Topic消费完后下线（零停服，但需要消费者同时订阅新旧Topic过渡期）

### 场景2：实时推荐系统（和你的项目最像）

**任务背景：**
用户在美团APP上浏览商家 → 点击菜品 → 加购 → 下单
这些行为实时发到Kafka → Flink/Spark消费 → 更新用户画像 → 推荐结果回写

**分配给候选人的任务：**
用户行为事件从APP端到推荐模型的全链路，要求：

- 端到端延迟 < 3秒（用户点了一个川菜馆，3秒内首页推荐要出现更多川菜）
- 日均10亿事件

**真实会遇到的问题：**

#### 问题3：消息体过大导致Broker OOM

- **现象：** 某天凌晨Broker集群3台机器接连OOM重启
- **原因：** 某个业务方往user-events里塞了一个新字段——用户的完整浏览历史列表（JSON数组），单条消息从平均500B膨胀到了800KB。`batch.size=32KB`的配置下，一个batch只能放一条消息，批量效果归零，内存分配暴涨。
- **解决方案：**

1. **紧急：** 在Producer端加消息大小校验，超过10KB直接拒绝并告警
2. Broker端配置 `message.max.bytes=1MB` 作为硬限制
3. **设计规范：** 大数据体（浏览历史）不走Kafka，写入对象存储/HBase，Kafka消息里只带引用ID
4. **代码review制度：** 修改Kafka消息schema必须经过架构评审

#### 问题4：消费者Rebalance风暴

- **现象：** 每隔几分钟就触发一次消费组rebalance，rebalance期间所有消费者停止消费，导致lag周期性飙高
- **原因：** 消费者处理一批消息耗时超过了 `max.poll.interval.ms`（默认5分钟），Kafka认为这个消费者"死了"，触发rebalance把它的分区分给别人。那个消费者处理完后回来，又触发一次rebalance...
- **解决方案：**

```properties
// 调整消费参数
max.poll.records=100          // 从默认500降到100，每次少拉点
max.poll.interval.ms=600000   // 从5分钟延长到10分钟
session.timeout.ms=30000      // 心跳超时保持30秒

```

- **根本解决：** 优化消费者处理逻辑。把重计算（推荐模型推理）从消费线程中拆出去，消费者只做"接消息 → 写入内部队列 → 提交offset"，重计算由独立线程池异步完成。

### 场景3：支付消息的精确一次语义

#### 问题5：消息重复消费导致重复扣款

- **现象：** 有用户投诉被扣了两次钱
- **原因：** 消费者处理完支付消息后，正要提交offset，此时这台机器挂了（JVM OOM）。重启后从上次提交的offset重新消费，导致这条支付消息被处理了两次。
- **解决方案：**

```java
// 方案A：幂等消费（最常用）
// 每条消息带唯一ID，消费前先查Redis/数据库看处理过没有
public void consume(PaymentEvent event) {
    String msgId = event.getMessageId();
    // Redis SETNX：如果key不存在则设置成功返回true
    if (!redis.setIfAbsent("consumed:" + msgId, "1", 24h)) {
        log.warn("重复消息，跳过: {}", msgId);
        return;  // 已处理过，直接跳过
    }
    paymentService.process(event);
}

// 方案B：事务性消费（更强，但更复杂）
// 把"业务处理"和"offset提交"放在同一个数据库事务里
@Transactional
public void consume(PaymentEvent event) {
    // 1. 写入业务表
    paymentService.process(event);
    // 2. 把offset写入同一个数据库（而不是提交给Kafka）
    offsetStore.save(topic, partition, offset);
    // 事务保证：要么都成功，要么都回滚
}

```

## 三、面试追问链（鉴别真伪的杀手锏）

一条完整的追问链，越往后越能暴露真伪：

- **Q1: 你们用Kafka做什么？**
- A: 用户行为事件采集，发到Flink做实时推荐

- **Q2: 日均多大量？峰值多少？**
- A: 日均2亿，峰值每秒3万 ← 说不出具体数字的基本没做过

- **Q3: 分区数多少？消费者几个实例？**
- A: 18个分区，6个消费者实例 ← 开始考验记忆力了

- **Q4: 为什么是18个分区？调整过吗？**
- A: 最初12个，后来...(省略) ← 能说出调整过程的一定做过

- **Q5: 出过什么生产事故？**
- A: 有一次消息积压...(省略) ← 没经历过事故的说明没上过生产

- **Q6: 你们怎么监控Kafka的？lag告警阈值设多少？**
- A: 用Kafka Manager + Prometheus，lag阈值10万 ← 这是日常运维才知道的

- **Q7: 如果让你重新设计，你会改什么？**
- A: 我会把行为事件和交易事件分开Topic... ← 有反思说明真正理解了

> Q5和Q6是分水岭——背八股的人到这里基本编不下去了，因为事故细节和监控运维是无法从博客里背到的。

## 四、给你（作为学习者）的建议

你现在的项目是很好的实战素材，但要让面试官认可，你需要做到：

1. **能说出具体数字**
   "我的Topic有12个分区，3个副本" ← 而不是"若干个分区"
2. **能说出踩过的坑**
   "我最开始acks=1，后来压测发现停broker会丢消息，改成了all"
3. **能说出调优过程**
   "batch.size从16KB调到32KB，吞吐量提升了40%，但P99延迟从8ms涨到15ms"
4. **能说出监控手段**
   "我用kafka-consumer-groups.sh看lag，用JMX导出指标到Grafana"

本质上面试官考的不是知识，是经验的密度。同样说"我用过Kafka"，说出3个生产事故和解决方案的人，和只能背出概念的人，面试官一听就能区分。
