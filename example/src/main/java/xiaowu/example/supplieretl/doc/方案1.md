基于当前代码现状，推荐不要直接改造现有 `SupplierConnection`，而是平行新增一套“通用数据源接入”能力。原因是现在的连接模型本质上是“供应商调度连接”，不是“通用数据源连接”，这一点可以从 [SupplierConnection.java](/c:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/domain/entity/SupplierConnection.java) 和 [schema.sql#L76](/c:/Users/admin/Desktop/recommendation_system/example/src/main/resources/sql/schema.sql#L76) 看出来；而前端预览页虽然已经有“数据源配置 / 测试连接”的展示，但目前还是静态页面，[supplier-etl-platform-preview.html#L584](/c:/Users/admin/Desktop/recommendation_system/example/src/main/resources/static/supplier-etl-platform-preview.html#L584) 这个按钮还没有后端联动。当前项目已有 Kafka/Redis 基础依赖，但默认库仍是 H2，[application.yml#L8](/c:/Users/admin/Desktop/recommendation_system/example/src/main/resources/application.yml#L8) 和 [application.yml#L27](/c:/Users/admin/Desktop/recommendation_system/example/src/main/resources/application.yml#L27) 也说明了现在主要是本地演示链路。

**推荐方案**
采用“新增通用接入子模块”的方式，挂在 `supplieretl` 下，但与现有 supplier 调度链路解耦。

1. 新增通用连接模型
   新建 `DataSourceConnection`、`DataSourceType`、`ConnectionTestResult`，支持 `KAFKA`、`MYSQL`、`REDIS`、`EXCEL`。
2. 新增连接管理表
   建议新增 `etl_data_source_connection`、`etl_connection_test_log`、`etl_ingestion_job` 三张表，避免污染现有 `supplier_connection`。
3. 新增统一测试接口
   提供 REST API：新增连接、查询连接、测试连接、执行一次导入。
4. 按类型实现测试器
   Kafka 用 `AdminClient` 测 broker/topic；MySQL 用独立 JDBC 连接测 `SELECT 1`；Redis 用独立连接测 `PING`；Excel 用上传文件后的解析校验替代“连接测试”。
5. 新增接入执行器
   Kafka 支持按 topic 拉取测试批次；MySQL 支持按 SQL/表读取；Redis 支持按 key pattern/structure 读取；Excel 支持 sheet/header/前 N 行预览。
6. 后续再接前端
   把静态预览页改成真正调用后端接口。

**为什么不建议复用现有 `SupplierConnection`**

1. 现有模型带有 `pullIntervalSeconds`、`nextPullAt`、`leaseUntil`、`retryCount` 这些调度语义，不适合 Redis/MySQL/Excel 这种通用接入。
2. 现有原始事件 `RawDataEvent` 也是 supplier 领域字段，不适合直接承载通用数据源。
3. 如果强行复用，后面 Excel、文件上传、云数据库连接参数都会把 supplier 领域搅乱。

**实施计划**

1. 第一阶段：连接管理与连接测试
   目标：先把“可配置、可测试”做通。
   交付：新表、新实体、新仓储、新 controller、新 service；支持 Kafka/MySQL/Redis/Excel 四类连接配置；支持本地 Kafka、云 MySQL 测试；保存最近一次测试结果和错误信息。
2. 第二阶段：数据接入执行
   目标：把“测试通过”升级成“可导入”。
   交付：Kafka topic 拉取、MySQL 查询导入、Redis key 读取、Excel 上传解析；统一写入平台原始接入表或原始事件表。
3. 第三阶段：接入预览页
   目标：把静态页改成真页面。
   交付：连接列表、表单、测试按钮、导入按钮、测试结果回显。
4. 第四阶段：完善测试与文档
   目标：保证后续可维护。
   交付：单元测试、集成测试、使用说明、示例配置。
