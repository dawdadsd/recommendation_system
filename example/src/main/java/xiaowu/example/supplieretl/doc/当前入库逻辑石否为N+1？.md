不是经典意义上的 N+1，但有“逐条写库导致的多次往返”问题。

经典 N+1 一般是：
`先查 1 次主表 + 再按每条结果各查 1 次子表`
这是 ORM 懒加载里最常见的读放大。

你这里在 [JdbcNormalizedSupplierRecordRepository.java](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/persistence/JdbcNormalizedSupplierRecordRepository.java#L65) 不是这个模式，因为没有“先查一批、再逐条补查”。
真正的成本在 [upsertOne(...)](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/persistence/JdbcNormalizedSupplierRecordRepository.java#L83) 里，它是“每条 record 至少 1 次、最多 3 次 SQL”：

- 已存在且可更新：`1` 次 `UPDATE`
- 不存在：`1` 次失败 `UPDATE` + `1` 次 `INSERT`，共 `2` 次
- 并发冲突/重复键：`UPDATE` + `INSERT` 失败 + 重试 `UPDATE`，最多 `3` 次

所以它更准确叫：
`row-by-row upsert` 或 `write amplification`
不是 N+1，但确实是 O(N) 次数据库往返，批量大时会慢。

我当时这样写，是为了先满足这几个目标：

- 不先 `SELECT` 判断存在，避免真的变成“每条先查再写”
- 保持 H2 / MySQL 这类环境都能跑，不强绑某个方言
- 顺手把“旧事件不能覆盖新事件”这个规则塞进 `UPDATE ... last_pulled_at <= ?` 里

所以这是一个“优先正确性和兼容性”的版本，不是最终性能最优版本。

如果你问生产上要不要优化，答案是：要看量级。

- 如果每次 ERP page size 很小，比如几十到一两百条，这种写法能接受，逻辑也直观
- 如果一批几千上万，或者 consumer 吞吐要很高，这里就该优化了

更优方向有 3 个：

1. 先在内存里按 `(supplierId, erpType, sourceRecordId)` 去重，只保留最新 `pulledAt`
2. 改成数据库原生 upsert
   如果生产库就是 MySQL，可以用 `INSERT ... ON DUPLICATE KEY UPDATE`
3. 用 `batchUpdate`
   就算还保留两段式策略，也能减少 JDBC 往返

所以结论是：

- `switch` 本身没问题
- `for + upsertOne` 不属于经典 N+1
- 但它确实存在“逐条 upsert 的写放大”问题
- 当前版本偏“教学/兼容/安全”，后续如果走生产化，建议我把它改成“批量 + 原生 upsert”版
