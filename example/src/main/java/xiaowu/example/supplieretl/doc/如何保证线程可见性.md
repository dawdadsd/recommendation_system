`volatile` 不是唯一能保证线程可见性的手段。

**先把 3 个词分开**

- 可见性：线程 A 改了值，线程 B 能看到新值
- 原子性：一个操作不会做一半被别人插进来
- 有序性：指令不会被乱重排到破坏语义

**除了 `volatile`，还能靠什么保证可见性**

- `synchronized`
  退出同步块时会把修改刷新出去，进入同步块时会看到之前的结果
- `Lock` / `ReentrantLock`
  `unlock()` 和后续 `lock()` 之间也有可见性保证
- `AtomicInteger`、`AtomicReference` 等原子类
  它们底层本质上也是 `volatile` + CAS
- `Thread.start()` / `Thread.join()`
  启动前的写，对新线程可见；线程结束前的写，在 `join()` 后可见
- 并发工具类
  比如 `CountDownLatch`、`Future.get()`、`BlockingQueue`，都自带 happens-before 语义
- `final` 字段
  对象构造完成且“安全发布”后，其他线程能正确看到 `final` 字段的值
- `VarHandle` / 内存屏障
  更底层，更灵活，也更容易用错

**JMM 是什么**
JMM 是 `Java Memory Model`，Java 内存模型。

它规定的是：

- 多线程下共享变量怎么读写才算合法
- 哪些重排序允许，哪些不允许
- 什么情况下一个线程的写，对另一个线程一定可见

你可以把 JMM 理解成 Java 并发的“交通规则”。
最核心的关键词是 `happens-before`。

例如：

- 对一个 `volatile` 变量的写，happens-before 后续对它的读
- `unlock` happens-before 后续 `lock`
- 线程里的操作 happens-before 另一个线程成功 `join` 之后的读取

**JNI 是什么**
JNI 是 `Java Native Interface`，Java 调本地代码的接口。

它用来让 Java 调：

- C / C++
- 操作系统原生库
- 已有的高性能 native 库
- 驱动、图像、音视频、硬件接口之类

它和 JMM 不是一回事：

- JMM：Java 并发内存规则
- JNI：Java 和 native 代码互通的桥

**什么时候用 `volatile`**
适合：

- 一个线程写，多个线程读的状态标志
  例：`shutdown`, `started`, `ready`
- 发布一个新的不可变对象引用
  例：`volatile Config currentConfig`
- 双重检查单例里的实例引用
  例：`private static volatile Singleton instance`

不适合：

- `count++`
- `if (x == 0) x = 1`
- 多个字段要保持一致
- 需要“读-改-写”整体原子性的场景

**一句话记**

- `volatile`：保证可见性 + 一定程度的有序性
- `volatile`：不保证复合操作的原子性

比如这段就不安全：

```java
private volatile int count;
count++;
```

因为 `count++` 不是原子操作。
