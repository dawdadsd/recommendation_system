**Spring 启动时，它怎么一步一步装起来**

1. Spring 扫描到这个配置类，因为它有 `@Configuration`，见 [SupplierRuntimeConfiguration.java#L29](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/config/SupplierRuntimeConfiguration.java#L29)。

2. `@EnableConfigurationProperties(...)` 把配置文件里的值绑定成 Bean，见 [SupplierRuntimeConfiguration.java#L31](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/config/SupplierRuntimeConfiguration.java#L31)。

   - [SupplierWorkerProperties.java#L5](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/config/SupplierWorkerProperties.java#L5)

   它们最终绑定的是 [application.yml#L80](C:/Users/admin/Desktop/recommendation_system/example/src/main/resources/application.yml#L80) 这一段配置。

3. `@EnableScheduling` 打开定时任务开关，见 [SupplierRuntimeConfiguration.java#L30](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/config/SupplierRuntimeConfiguration.java#L30)。
   这样 [SupplierPullSchedulingJob.java#L24](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/scheduling/SupplierPullSchedulingJob.java#L24) 上的 `@Scheduled` 才会真正执行。

4. Spring 执行这个类里的 `@Bean` 方法，创建运行时需要的核心对象。
   例如：
   - `kingdeeErpAdapter(...)` 需要 `KingdeeErpProperties + ObjectMapper`
   - `yonyouErpAdapter(...)` 需要 `YonyouErpProperties + ObjectMapper`
   - `routingSupplierPullClient(...)` 需要前两个 adapter 再加一个 fallback `MockSupplierPullClient`
   - `supplierPullExecutionApplicationService(...)` 再依赖 repo、pull client、publisher、audit repo
   - `supplierRawDataProcessingApplicationService(...)` 再依赖 parser dispatcher、标准化仓储、raw publisher

5. 其他不是这里创建的 Bean，来自别的地方：
   - 调度应用服务来自 [SupplierSchedulingApplicationConfiguration.java#L13](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/config/SupplierSchedulingApplicationConfiguration.java#L13)
   - Kafka topic 和 publisher 来自 [SupplierKafkaConfiguration.java#L19](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/kafka/config/SupplierKafkaConfiguration.java#L19)
   - repository / consumer / parser 多数是 `@Repository`、`@Component` 自动扫描出来的

所以你可以把 Spring 容器理解成在做一件事：

“先把配置对象准备好，再把基础设施 Bean 准备好，最后把应用服务拼起来。”

**为什么不直接都写成 `@Service`**

这是这套代码风格里很值得学的一点。

像 [SupplierPullExecutionApplicationService.java](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/application/service/SupplierPullExecutionApplicationService.java) 和 [SupplierRawDataProcessingApplicationService.java](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/application/service/SupplierRawDataProcessingApplicationService.java) 都没有直接标 `@Service`，而是由配置类 `new` 出来。

这样做的好处是：

- 应用层不直接依赖 Spring 注解，领域/应用代码更干净
- 依赖关系更显式，打开配置类就能看到“这个服务到底吃哪些依赖”
- 测试更容易，直接 `new` 就能测，不必强依赖 Spring 上下文

代价是：

- 配置类会变成“装配中心”
- 如果 Bean 很多，配置类会越来越长，需要按职责拆分

你现在这个项目其实已经在拆了：

- 调度应用服务放在 [SupplierSchedulingApplicationConfiguration.java](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/config/SupplierSchedulingApplicationConfiguration.java)
- Kafka 放在 [SupplierKafkaConfiguration.java](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/kafka/config/SupplierKafkaConfiguration.java)
- 运行时执行链放在 [SupplierRuntimeConfiguration.java](C:/Users/admin/Desktop/recommendation_system/example/src/main/java/xiaowu/example/supplieretl/infrastructure/config/SupplierRuntimeConfiguration.java)

这个拆法是合理的。

**你可以怎么记这整个装配模型**

你可以把它记成四层：

1. 配置层
   `application.yml` + `@ConfigurationProperties`

2. 装配层
   `@Configuration` + `@Bean`

3. 运行入口层
   `@Scheduled` job、Kafka `@KafkaListener`

4. 业务执行层
   application service + repository + adapter

在这个 supplieretl 里，对应关系大致就是：
