# 供应商连接调度测试教学

下面这套步骤可以帮助你验证 `example` 模块里这条链路是否真的跑起来了：

```text
supplier_connection
  -> SupplierPullSchedulingJob / 手动 dispatch
  -> Kafka topic: supplier.pull.request
  -> SupplierPullTaskConsumer
  -> MockSupplierPullClient
  -> 回写 supplier_connection
```

## 1. 启动前说明

### 1.1 启动命令

```bash
mvn -pl example -DskipTests spring-boot:run
```

服务地址：

- `http://localhost:8924`

H2 控制台：

- `http://localhost:8924/h2-console`
- JDBC URL: `jdbc:h2:mem:exampledb`
- user: `sa`
- password: 空

### 1.2 Kafka 依赖

当前供应商调度链路会真的发 Kafka，所以本地需要有：

- Kafka broker：`localhost:9092`

如果 Kafka 没启动，应用可能无法正常完成调度演示。

### 1.3 自动调度默认开启

默认配置在：

- [application.yml](/c:/Users/admin/Desktop/recommendation_system/example/src/main/resources/application.yml)

默认值：

- `supplier.scheduling.enabled=true`
- `supplier.scheduling.fixed-delay-ms=10000`

这意味着应用启动后，每 10 秒会自动扫描一次到期供应商连接。

如果你想纯手动测试，可以先把：

```yaml
supplier:
  scheduling:
    enabled: false
```

改掉再启动。

## 4. 推荐测试顺序

### 4.1 先看 demo

PowerShell 下建议用 `curl.exe`，避免命中 PowerShell 自带别名：

```bash
curl.exe http://localhost:8924/api/examples/suppliers/demo
```

### 4.2 查看初始状态

```bash
curl.exe http://localhost:8924/api/examples/suppliers/connections
```

重点看这些字段：

- `status`
- `nextPullAt`
- `lastSuccessAt`
- `lastErrorAt`
- `lastCursor`
- `retryCount`
- `leaseUntil`
- `version`

### 4.3 手动触发一次调度

```bash
curl.exe -X POST http://localhost:8924/api/examples/suppliers/scheduling/dispatch ^
  -H "Content-Type: application/json" ^
  -d "{\"batchSize\":10,\"leaseSeconds\":30}"
```

如果你在 PowerShell 里更喜欢单行写法，也可以：

```bash
curl.exe -X POST "http://localhost:8924/api/examples/suppliers/scheduling/dispatch" -H "Content-Type: application/json" -d "{\"batchSize\":10,\"leaseSeconds\":30}"
```

### 4.4 再次查看状态

```bash
curl.exe http://localhost:8924/api/examples/suppliers/connections
```

你应该能观察到：

1. `SUPPLIER_ALPHA`
   - `lastSuccessAt` 被更新
   - `lastCursor` 被更新
   - `retryCount = 0`
   - `nextPullAt` 往后推进
2. `SUPPLIER_FAIL_ONCE`
   - 第一次会失败
   - `lastErrorAt` 被更新
   - `retryCount = 1`
   - `nextPullAt` 被推进到重试时间
3. `SUPPLIER_PAUSED`
   - 不会变化

### 4.5 查看单个供应商状态

```bash
curl.exe http://localhost:8924/api/examples/suppliers/connections/9001
```

```bash
curl.exe http://localhost:8924/api/examples/suppliers/connections/9002
```

## 5. H2 里怎么观察

进入 H2 控制台后，可以执行：

```sql
SELECT
    supplier_id,
    supplier_code,
    status,
    next_pull_at,
    last_success_at,
    last_error_at,
    last_cursor,
    retry_count,
    lease_until,
    version,
    updated_at
FROM supplier_connection
ORDER BY supplier_id;
```

这条 SQL 最适合和接口结果对照着看。

## 6. 如何重复演示

因为第一次调度后，`next_pull_at` 会被推进，所以第二次立刻手动触发，可能扫描不到任务。

如果你想重新演示，可以在 H2 里执行：

```sql
UPDATE supplier_connection
SET
    next_pull_at = TIMESTAMP '2026-04-02 15:00:00',
    lease_until = NULL
WHERE supplier_id IN (9001, 9002);
```

如果你想把 `SUPPLIER_FAIL_ONCE` 重置成“第一次失败”的效果，可以再执行：

```sql
UPDATE supplier_connection
SET
    retry_count = 0,
    last_error_at = NULL,
    lease_until = NULL,
    next_pull_at = TIMESTAMP '2026-04-02 15:00:00'
WHERE supplier_id = 9002;
```

然后重新调用：

```bash
curl.exe -X POST "http://localhost:8924/api/examples/suppliers/scheduling/dispatch" -H "Content-Type: application/json" -d "{\"batchSize\":10,\"leaseSeconds\":30}"
```

## 8. 一句话总结

现在这条教学链路已经能演示三件关键的事：

- 供应商调度不是直接调用外部 API，而是先扫描到期连接再发任务
- Kafka 在这里承担的是“调度后的执行队列”，不是时间调度器
- 成功、失败、重试和暂停，最后都要回写到 `supplier_connection` 这张调度真相表
