## 1. 系统概述

本系统是一个**分布式供应商数据拉取调度引擎**，核心职责是：

- **定时扫描**到期的供应商连接
- **分布式安全**地为每个供应商获取排他租约（lease）
- 通过 **Kafka** 将拉取任务异步化，解耦调度与执行
- **回写结果**：成功推进游标、失败触发重试策略

这个模式在互联网企业中广泛用于：电商商品同步、价格爬虫、数据集成管道（ELT）等场景。

### 2.1 分层结构（六边形架构）

```
┌───────────────────────────────────────────────────┐

│  SupplierPullSchedulingApplicationService         │  ← 调度：扫描+租约+发布
│  SupplierPullExecutionApplicationService          │  ← 执行：拉取+回写
│                                                   │
│  Port (接口约定):                                  │
│    PullTaskPublisher  SupplierPullClient          │
└──────┬────────────────────────────────┬────────────┘
       │                                │
┌──────▼──────────────┐    ┌────────────▼────────────┐
│    Domain Layer     │    │  Infrastructure Layer   │
│  SupplierConnection │    │  JdbcSupplierConn...    │ ← 数据库
│  (聚合根 + 状态机)   │    │  KafkaPullTaskPublisher │ ← Kafka 发布
│  ConnectionStatus   │    │  SupplierPullTaskConsumer│ ← Kafka 消费
│  Repository 接口     │    │  MockSupplierPullClient │ ← mock 执行
└─────────────────────┘    │  SupplierPullSchedulingJob│ ← @Scheduled
                           └─────────────────────────┘
```

### 2.2 调度与执行的分离原则

```
调度器（Scheduler）          执行器（Worker）
    ↓                             ↑
  扫描到期连接              消费 Kafka 消息
  获取排他租约              调用供应商 API
  发布任务到 Kafka           回写成功/失败结果
```

这是一个典型的 **Producer-Consumer 解耦**：调度器和执行器可以独立扩容，彼此感知失败但不阻塞。

## 3. 核心数据模型

### 3.1 supplier_connection 表字段语义

```sql
CREATE TABLE supplier_connection (
    supplier_id            BIGINT        PRIMARY KEY,   -- 业务主键
    supplier_code          VARCHAR(64)   NOT NULL,       -- 业务编码（唯一）
    status                 VARCHAR(32)   NOT NULL,       -- ACTIVE / PAUSED / DISABLED
    pull_interval_seconds  INT           NOT NULL,       -- 正常拉取间隔（秒）
    next_pull_at           TIMESTAMP     NOT NULL,       -- 下次允许调度的时间点 ★核心
    last_success_at        TIMESTAMP,                   -- 最后一次成功拉取时间
    last_error_at          TIMESTAMP,                   -- 最后一次失败时间
    last_cursor            VARCHAR(256),                 -- 增量拉取游标（分页锚点）
    retry_count            INT           NOT NULL,       -- 连续失败次数（成功后归0）
    lease_until            TIMESTAMP,                   -- 当前租约截止时间（NULL=空闲）★并发安全
    version                BIGINT        NOT NULL,       -- 乐观锁版本号 ★防竞争
    created_at             TIMESTAMP     NOT NULL,
    updated_at             TIMESTAMP     NOT NULL
);
```

**关键字段解读：**

| 字段           | 作用     | 设计意图                           |
| -------------- | -------- | ---------------------------------- |
| `next_pull_at` | 调度时钟 | 决定什么时候允许再次扫描此连接     |
| `lease_until`  | 分布式锁 | 防止多个调度节点同时处理同一供应商 |
| `version`      | 乐观锁   | UPDATE WHERE version=? 替代悲观锁  |
| `retry_count`  | 重试状态 | 驱动重试延迟策略                   |
| `last_cursor`  | 增量标记 | 支持断点续拉，避免全量重拉         |

### 3.2 状态机

```
              ┌──────────activate()──────┐
              ▼                          │
           ACTIVE ──pause()──► PAUSED ──┘
              │
              └──disable()──► DISABLED（终态，不可逆）
```

状态转换全部在 `SupplierConnection` 聚合根内完成，基础设施层不能绕过领域层直接改 status。

## 4. 完整调度链路详解

### 4.1 完整时序图

```
每10秒一次
SupplierPullSchedulingJob.scanAndDispatch()
        │
        ▼
SupplierPullSchedulingApplicationService.scheduleDueConnections(cmd)
        │
        ├─1. findSchedulableConnections(now, batchSize)
        │       SQL: WHERE status='ACTIVE'
        │             AND next_pull_at <= now
        │             AND (lease_until IS NULL OR lease_until <= now)
        │             ORDER BY next_pull_at ASC
        │             LIMIT batchSize
        │
        ├─2. for each candidate:
        │       tryAcquireLease(supplierId, version, now, leaseUntil, updatedAt)
        │               SQL: UPDATE ... WHERE supplier_id=? AND version=?
        │                    AND next_pull_at<=? AND (lease_until IS NULL OR lease_until<=?)
        │               ← 乐观锁，0行受影响=抢锁失败（被其他节点抢先）
        │
        ├─3. 抢锁成功后: publishPullRequested(event) → Kafka topic: supplier.pull.request
        │       key=supplierId (保证同一供应商消息有序)
        │       payload=JSON(supplierId, supplierCode, lastCursor, scheduledAt, leaseUntil)
        │
        └─4. Kafka publish 失败: releaseLease() → 回滚租约

Kafka Consumer (SupplierPullTaskConsumer.onMessage)
        │
        ├─1. 反序列化 PullRequestedEvent
        │
        ├─2. SupplierPullExecutionApplicationService.execute(command)
        │       │
        │       ├─ findBySupplierId(supplierId)  ← 重新从 DB 加载最新状态
        │       │
        │       ├─ supplierPullClient.pull(PullCommand)  ← 调用 mock / 真实供应商
        │       │
        │       ├─ 成功: markPullSuccess(supplierId, version, nextCursor, ...)
        │       │         SQL: UPDATE ... SET last_cursor=?, last_success_at=?,
        │       │                            retry_count=0, lease_until=NULL,
        │       │                            next_pull_at=?
        │       │                            WHERE supplier_id=? AND version=?
        │       │
        │       └─ 失败: markPullFailure(supplierId, version, retryCount+1, retryAt, ...)
        │                 SQL: UPDATE ... SET last_error_at=?, retry_count=?,
        │                                    lease_until=NULL, next_pull_at=?
        │                                    WHERE supplier_id=? AND version=?
        │
        └─3. 成功/失败日志
```

### 4.2 乐观锁竞争场景

当两个调度节点 A、B 同时扫描到 supplierId=9001：

```
时间线:
  T1: A 读取 version=5
  T1: B 读取 version=5
  T2: A 执行 UPDATE ... WHERE version=5 → 成功，version变成6
  T2: B 执行 UPDATE ... WHERE version=5 → 失败（0行受影响）→ 跳过
```

这是**乐观锁的核心价值**：无需数据库行锁或 Redis 锁，利用 CAS（Compare and Swap）保证并发安全。

### 4.3 租约（Lease）的作用

租约解决的是"调度成功但 Worker 崩溃"的问题：

```
正常流程: Scheduler → lease 30秒 → Worker 执行完成 → lease 清除（markPullSuccess/Failure）

异常流程（Worker 崩溃）:
  T0: lease_until = now + 30s
  T30: Worker 崩溃，lease 未清除
  T31: 下一轮 Scheduler 扫描，发现 lease_until <= now → 重新调度
```

**自愈性**：无需人工干预，租约超时后系统自动重新调度。

## 5. 关键算法分析

### 5.1 当前重试延迟算法（线性固定延迟）

```java
// SupplierPullExecutionApplicationService.java
connection.markPullFailed(
    errorAt,
    connection.calculateRetryAtFrom(errorAt, command.retryDelaySeconds())
);

// SupplierConnection.java
public LocalDateTime calculateRetryAtFrom(LocalDateTime baseTime, int retryDelaySeconds) {
    return baseTime.plusSeconds(retryDelaySeconds);  // 固定 45 秒
}
```

**问题**：无论第几次失败，都是固定 45 秒后重试。

### 5.2 当前调度查询算法

```sql
-- findSchedulableConnections
WHERE status = 'ACTIVE'
  AND next_pull_at <= ?
  AND (lease_until IS NULL OR lease_until <= ?)
ORDER BY next_pull_at ASC        -- FIFO，最早应该拉取的优先
LIMIT ?
```

**问题**：所有 ACTIVE 供应商同等优先级，没有基于重要性或 SLA 的差异化。

### 5.3 当前批量处理算法

```java
// SupplierPullSchedulingApplicationService.java
for (SupplierConnection connection : candidates) {
    boolean acquired = tryAcquireLease(...);   // 串行逐个抢锁
    if (!acquired) continue;
    publishPullRequested(event);               // 串行逐个发布
}
```

**问题**：串行处理，batchSize=50 时每次循环需要 50 次数据库 UPDATE + 50 次 Kafka send。

## 6. 现有算法的瓶颈

| 问题               | 影响                                                                       | 严重程度 |
| ------------------ | -------------------------------------------------------------------------- | -------- |
| 固定重试延迟       | 雪崩时所有失败供应商同时重试，造成流量尖峰                                 | ★★★★☆    |
| 无退避上限         | retryCount 持续增加但延迟不变，频繁打扰不可用供应商                        | ★★★★☆    |
| 串行 Kafka publish | batchSize 大时调度延迟高                                                   | ★★★☆☆    |
| 无优先级区分       | 高价值供应商和低价值供应商同等排队                                         | ★★★☆☆    |
| 无死信处理         | Kafka 消费异常会抛出异常触发 Kafka 内置重试，但poison message 可能无限循环 | ★★★★★    |
| 无租约主动清理     | 依赖下次扫描才发现过期租约，最多延迟一个调度周期                           | ★★☆☆☆    |

## 7. 算法优化方案

### 7.1 指数退避 + 随机抖动（最高优先）

**核心公式：**

$$
\text{retryDelay} = \min\left(\text{maxDelay},\ \text{baseDelay} \times 2^{\text{retryCount}}\right) + \text{random}(0,\ \text{jitterMs})
$$

**效果对比：**

| retryCount | 当前（固定45s） | 优化后（base=30s, max=3600s） |
| ---------- | --------------- | ----------------------------- |
| 0          | 45s             | 30s + jitter                  |
| 1          | 45s             | 60s + jitter                  |
| 2          | 45s             | 120s + jitter                 |
| 3          | 45s             | 240s + jitter                 |
| 5          | 45s             | 960s ≈ 16分钟                 |
| 10         | 45s             | 3600s = 1小时（封顶）         |

```
随机抖动的意义：防止"惊群效应"
假设 1000 个供应商同时失败:
  - 无抖动: 45s 后 1000 个同时重试 → 瞬间流量峰值
  - 有抖动: 在 [45s, 60s] 之间均匀散开 → 流量平滑
```

### 7.2 优先级调度（Priority Scheduling）

在当前 FIFO 查询基础上，引入**优先级字段**：

```sql
-- 方案A: 新增 priority 列
ALTER TABLE supplier_connection ADD COLUMN priority INT NOT NULL DEFAULT 5;

-- 优先按 priority 高的先调度，再按 next_pull_at
ORDER BY priority DESC, next_pull_at ASC
```

或基于**动态评分**（不修改表结构）：

```sql
-- 方案B: 动态计算 score（retryCount 越多排越后，越老的 next_pull_at 排越前）
ORDER BY
    CASE WHEN retry_count > 5 THEN 1 ELSE 0 END ASC,  -- 高重试次数降级
    next_pull_at ASC
```

### 7.3 并行 Lease 获取（提升吞吐）

当前串行流程耗时：`N × (DB_RTT + Kafka_RTT)`
优化后并行流程耗时：`max(DB_RTT) + max(Kafka_RTT)`

优化方向：将 Kafka publish 改为异步发送（不调用 `.join()`），但保留 lease release 的容错逻辑。

### 7.4 死信队列（DLQ）处理

当前：消费失败 → 抛出异常 → Kafka 重试（可能无限循环）
优化：超过 N 次重试后，将 poison message 转入 DLQ topic，避免阻塞正常任务。

### 7.5 自适应拉取间隔

基于历史成功率动态调整 `pullIntervalSeconds`：

```
成功率 95%+ → 可缩短间隔（加快拉取频率）
成功率 60%- → 自动拉长间隔（降低对供应商的压力）
```

## 8. 优化实现代码

### 8.1 指数退避算法（替换 SupplierConnection.calculateRetryAtFrom）

```java
/**
 * 指数退避 + 随机抖动的重试延迟计算。
 *
 * 公式: delay = min(maxDelaySeconds, baseDelaySeconds * 2^retryCount) + random(0, jitterMs)
 *
 * @param baseTime          失败的基准时间
 * @param retryCount        当前连续失败次数（调用前已 +1）
 * @param baseDelaySeconds  基础延迟秒数（建议 30）
 * @param maxDelaySeconds   最大延迟秒数（建议 3600，即1小时封顶）
 * @param maxJitterMs       随机抖动最大毫秒数（建议 10000，即10秒）
 */
public LocalDateTime calculateRetryAtWithBackoff(
        LocalDateTime baseTime,
        int retryCount,
        int baseDelaySeconds,
        int maxDelaySeconds,
        int maxJitterMs) {

    Objects.requireNonNull(baseTime, "baseTime must not be null");
    if (baseDelaySeconds <= 0) throw new IllegalArgumentException("baseDelaySeconds must be positive");
    if (maxDelaySeconds <= 0) throw new IllegalArgumentException("maxDelaySeconds must be positive");

    // 指数退避：base * 2^retryCount，使用 Math.min 防止溢出
    long exponentialDelay = (long) baseDelaySeconds * (1L << Math.min(retryCount, 30));
    long cappedDelay = Math.min(exponentialDelay, maxDelaySeconds);

    // 随机抖动：ThreadLocalRandom 线程安全，性能优于 Random
    long jitterMs = ThreadLocalRandom.current().nextLong(0, Math.max(1, maxJitterMs));

    return baseTime
            .plusSeconds(cappedDelay)
            .plusNanos(jitterMs * 1_000_000L);
}
```

### 8.2 调度服务层集成退避

在 `SupplierPullExecutionApplicationService` 中替换固定延迟调用：

```java
// 失败时，使用指数退避代替固定延迟
LocalDateTime errorAt = LocalDateTime.now();
LocalDateTime retryAt = connection.calculateRetryAtWithBackoff(
        errorAt,
        connection.getRetryCount() + 1,  // 失败后 retryCount 是更新前的值
        30,       // base: 30秒
        3600,     // max: 1小时
        10_000    // jitter: 最大10秒随机抖动
);
connection.markPullFailed(errorAt, retryAt);
```

### 8.3 优先级降级 SQL（无需改表结构）

```sql
-- 替换 findSchedulableConnections 的 ORDER BY
ORDER BY
    -- 高重试次数（>= 5次）降级到队列末尾
    CASE WHEN retry_count >= 5 THEN 1 ELSE 0 END ASC,
    -- 其余按 next_pull_at 时间顺序（最早到期的最优先）
    next_pull_at ASC,
    supplier_id ASC
LIMIT ?
```

### 8.4 Kafka 死信处理（Consumer 侧）

```java
@KafkaListener(topics = "${supplier.kafka.topic.pull-request}",
               errorHandler = "supplierPullErrorHandler")
public void onMessage(String payload) {
    // ... 原有逻辑
}

// 注册死信处理 Bean
@Bean
public KafkaListenerErrorHandler supplierPullErrorHandler(
        KafkaTemplate<String, String> kafkaTemplate,
        SupplierKafkaProperties properties) {
    return (message, exception) -> {
        // 超过3次重试（DeadLetterPublishingRecoverer 配置）后转入 DLQ
        String dlqTopic = properties.topic().pullRequest() + ".DLQ";
        kafkaTemplate.send(dlqTopic, String.valueOf(message.getPayload()));
        log.error("[SupplierWorker] Sending to DLQ, origin payload={}", message.getPayload(), exception);
        return null;
    };
}
```

## 9. 性能对比

### 9.1 重试策略对比（1000个供应商同时失败场景）

```
当前方案（固定45s）：
  T+0s:  1000次失败
  T+45s: 1000个 next_pull_at 到达 → 下一轮调度扫出1000个 → Kafka 峰值
  T+90s: 再次 1000个到达 → 再次峰值

优化方案（指数退避+jitter，base=30s，jitter=10s）：
  T+0s:   1000次失败
  T+30~40s: ~1000/10s × 10秒窗口 = 均匀分散
  失败2次的供应商: 60~70s 后
  失败5次的供应商: 960s+ 后，已基本不影响正常供应商的调度
```

### 9.2 调度不公平问题（无优先级 vs 有优先级）

```
无优先级方案：
  SUPPLIER_FAIL_50_TIMES (retryCount=50) 和 SUPPLIER_ALPHA (retryCount=0)
  同等竞争调度资源。retryCount=50 的供应商大概率已不可用，却占用调度 slot。

有优先级（重试降级）：
  retryCount >= 5 的供应商降级到队尾
  正常供应商优先得到调度，整体系统健康度更高
```

### 9.3 磁盘/网络 I/O 分析

```
当前每轮调度（batchSize=50）的 I/O：
  1 × SELECT（扫描候选）
  50 × UPDATE tryAcquireLease（串行）
  50 × Kafka send().join()（同步等待 ack）

优化后（Kafka 异步发送）：
  1 × SELECT
  50 × UPDATE tryAcquireLease（可以并行，但 H2 单线程需谨慎）
  50 × Kafka send()（异步，不等待 ack，通过回调处理 lease release）
  预估 Kafka 侧延迟下降 60~80%
```

## 小结

| 优化点                  | 实现难度                    | 效果                       | 推荐优先级 |
| ----------------------- | --------------------------- | -------------------------- | ---------- |
| 指数退避 + jitter       | 低（改 1 个方法）           | 防雪崩，显著降低峰值压力   | P0 立刻做  |
| 重试次数多的降级排队    | 低（改 1 条 SQL）           | 提升正常供应商的调度公平性 | P1         |
| Kafka DLQ 死信处理      | 中（新增 Bean 配置）        | 防 poison message 无限阻塞 | P1         |
| Kafka 异步发送          | 中（需改 Publisher + 回调） | 降低调度器延迟             | P2         |
| 自适应拉取间隔          | 高（需要统计历史成功率）    | 进一步降低对供应商的压力   | P3         |
| 优先级字段 + 表结构变更 | 高（DDL + 业务分类逻辑）    | 支持 SLA 差异化            | P3         |
