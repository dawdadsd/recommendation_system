## 核心问题

你现在的 `ChatService` 是串行的：查画像 → 拼 Prompt → 调 AI。如果用户问"推荐一款手机"，你其实需要同时查多个数据源：

```text
用户消息: "推荐一款手机"
│
├─ 查用户画像 (50ms)   ← 这三个可以并行
├─ 查行为记录 (80ms)
├─ 查商品目录 (60ms)
│
▼ 串行等最慢的: 80ms（而非 50+80+60=190ms）
│
拼装上下文 → AI 回复
```

**关键设计问题有三个：**

### 问题 1：AI 怎么知道该调哪些工具？

两种路线：

| 路线            | 做法                                          | 优缺点                     |
| :-------------- | :-------------------------------------------- | :------------------------- |
| **A. 意图路由** | 先用关键词/规则判断用户意图，再决定调哪些工具 | 简单可控，但规则写死不灵活 |
| **B. 全量预取** | 每次都并发查所有数据源，统一注入上下文        | 不需要意图判断，但查询量大 |

我建议 **Phase 1 用方案 B（全量预取）**。原因：你的数据源只有 3 个（画像、行为、商品），都是走索引的单表查询，总耗时 < 100ms。预取成本极低，但省掉了意图识别的复杂度。

### 问题 2：并发模型怎么选？

Java 21+ 有三种选择：

```java
// 方案 1: ExecutorService + Virtual Threads（推荐，最简单）
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Future<A> f1 = executor.submit(() -> queryProfile(userId));
    Future<B> f2 = executor.submit(() -> queryBehavior(userId));
    Future<C> f3 = executor.submit(() -> queryItems(userId));
    // 等待全部完成
    var profile = f1.get(3, TimeUnit.SECONDS);
    var behavior = f2.get(3, TimeUnit.SECONDS);
    var items = f3.get(3, TimeUnit.SECONDS);
}

// 方案 2: StructuredTaskScope（Java 21 Preview，更优雅但需要 --enable-preview）
// 方案 3: CompletableFuture（回调地狱，不推荐）
```

推荐**方案 1**：`newVirtualThreadPerTaskExecutor`。不需要 preview flag，虚拟线程开销几乎为零，代码直观。

### 问题 3：工具的抽象层怎么设计？

定义一个统一的 `Tool` 接口，每个数据源是一个 `Tool` 实现：

```java
// 工具执行结果
public record ToolResult(String toolName, String context) {}

// 工具接口
public interface ChatTool {
    String name();
    ToolResult execute(Long userId);
}
```

这样未来加新工具（比如查订单历史、查优惠券），只需要实现接口，注册到 Spring 容器即可。

### 先理解本质：AI 交互到底是什么？

抛开所有框架，AI 对话的本质就是 **一次 HTTP POST 请求**：

```text
你的应用 ──── HTTP POST ────► OpenAI API
 │                          │
 │ Request Body:            │
 │ {                        │
 │   "model": "gpt-4o-mini",
 │   "messages": [          │
 │     { "role": "system", "content": "你是..." },
 │     { "role": "user", "content": "推荐手机" }
 │   ],                     │
 │   "tools": [...]         │ ← 告诉 AI 有哪些工具可用
 │ }                        │
 │                          │
 ◄── JSON Response ───┘
 {
   "choices": [{
     "message": {
       "role": "assistant",
       "content": "推荐 iPhone 16...",
       "tool_calls": [...]    ← AI 决定调用哪些工具
     }
   }]
 }
```

Spring AI 做的事情，就是把这个 HTTP 调用包装成 Java 对象。没有魔法。

### 工具调用的核心循环

这是整个 AI 工具调用最精髓的部分 —— 一个多轮对话循环：

- **第 1 轮：**
  - 你 → AI: "推荐一款手机" + tools定义`[queryUserProfile, searchItems]`
  - AI → 你: "我需要调用 `queryUserProfile(userId=1)` 和 `searchItems(keyword=手机)`"
  - _（AI 不回答问题，而是返回 `tool_calls`）_

- **第 2 轮：**
  - 你 → AI: "这是工具执行结果：
    - `queryUserProfile` → `{偏好: 3C数码, 预算: 2000-5000}`
    - `searchItems` → `[{iPhone 16, ¥5999}, {Pixel 9, ¥4999}]`"
  - AI → 你: "根据你的偏好和预算，推荐 Pixel 9..."
  - _（AI 拿到工具结果后，生成最终回答）_

这就是 Spring AI 的 `FunctionCallbackWrapper` 在背后做的事情。 它把这个循环藏起来了，让你以为 AI 自己就能查数据库。实际上是：**AI 告诉你要调什么 → 你执行 → 结果喂回去 → AI 再回答**。

### 我们要造什么？

```text
xiaowu.backed.infrastructure.ai/
├── model/                     // 数据模型（对应 OpenAI JSON 协议）
│   ├── ChatMessage.java       // { role, content, toolCalls }
│   ├── ChatRequest.java       // 发给 AI 的请求体
│   ├── ChatResponse.java      // AI 返回的响应体
│   ├── ToolCall.java          // AI 要求调用的工具
│   └── ToolDefinition.java    // 工具的 JSON Schema 描述
├── client/
│   └── OpenAiClient.java      // 原始 HTTP 调用（Java HttpClient）
├── memory/
│   └── ConversationMemory.java// 对话记忆（消息历史管理）
├── tool/
│   ├── ChatTool.java          // 工具接口
│   └── ChatToolExecutor.java  // 虚拟线程并发执行器
└── core/
    └── AiChatEngine.java      // 编排引擎（核心循环）
```

**每一层对应 Spring AI 的什么：**

| 我们造的             | Spring AI 对应     | 本质                         |
| :------------------- | :----------------- | :--------------------------- |
| `OpenAiClient`       | `OpenAiApi`        | HTTP POST + JSON 序列化      |
| `ChatMessage`        | `Message` 接口     | role + content 的载体        |
| `ConversationMemory` | `ChatMemory`       | `List<Message>` 的管理       |
| `ChatTool`           | `FunctionCallback` | 工具定义 + 执行逻辑          |
| `ChatToolExecutor`   | _Spring AI 没有_   | **我们的创新：虚拟线程并发** |
| `AiChatEngine`       | `ChatClient`       | 编排整个调用循环             |

### 开发顺序

我建议按这个顺序，每一步都能独立测试：

- **Phase 1: 能发消息、收回复（最小可用）**

  - → **测试:** AI 能记住上一轮说了什么
- **Phase 3: 工具调用（核心）**
  - → `ToolDefinition` + `ToolCall` + `ChatTool` + `ChatToolExecutor`
  - → `AiChatEngine` 实现工具调用循环
  - → **测试:** 用户问推荐 → AI 自动调工具查数据库 → 返回个性化结果
- **Phase 4: 流式输出（SSE）**
  - → 改造 `OpenAiClient` 支持 `stream: true`
  - → **测试:** 回复逐字输出，而非等全部完成
