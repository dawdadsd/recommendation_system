想清楚了。核心结论是：多路召回的融合不要做成一个“全局固定比例”，而要做成“按用户状态动态配比 + 统一归一化打分 + 排序兜底”。

我建议先用这套比例起步：

1. 新用户/冷启动

- 热门召回：60%
- 内容相似召回：30%
- 近期行为召回：10% 或 0%
- 理由：行为少，热门负责保底，内容相似负责个性化起步。

2. 轻度活跃用户

- 热门召回：30%
- 近期行为召回：40%
- 内容相似召回：30%
- 理由：开始有行为，但还不够稳定，不能让近期行为吃满。

3. 稳定活跃用户

- 热门召回：15%
- 近期行为召回：55%
- 内容相似召回：30%
- 理由：近期行为最能反映短期兴趣，热门只做探索和兜底。

4. 兴趣很聚焦的深度用户

- 热门召回：10%
- 近期行为召回：45%
- 内容相似召回：45%
- 理由：这类用户更适合“延续兴趣 + 相邻扩展”，热门占比要压低。

落地时不要只看“数量比例”，还要同时做“分数融合”：

```text
fusionScore = w_hot * hotScoreNorm
            + w_behavior * behaviorScoreNorm
            + w_content * contentScoreNorm
            + multiSourceBonus
            + freshnessBonus
```

这里我建议：

- `hotScoreNorm`、`behaviorScoreNorm`、`contentScoreNorm` 都用 rank-based 归一化，不要直接混原始分数。
- 同一个 item 被多路同时召回时，加 `multiSourceBonus`，因为这通常代表更强置信度。
- 热门召回只负责“保底和探索”，不要在融合分里权重过大，不然会把系统做成热门榜。

工程上建议这样分层：

- 召回层：每路先各自取 TopK。
- 融合层：去重、归一化、按权重合并。
- 重排层：再把融合后的 Top200/Top500 交给排序模型。
- 解释层：记录 item 来自哪些召回路由，例如“因为你最近看过 X”或“当前热门”。

如果你要一个更实用的初版，我建议直接定成：

- 冷启动：`6:0:4` 或 `6:1:3`
- 普通用户：`3:4:3`
- 活跃用户：`1.5:5.5:3`
  也就是热门 : 近期行为 : 内容相似。

再往前一步，版本治理也要一起设计：

- 融合比例单独做成“召回策略版本”，不要和模型版本绑死。
- 支持按版本灰度发布。
- 回滚时只切换“生效中的融合策略版本”，不要重新训练模型。
- 清理时保留最近稳定版本 + 当前灰度版本 + 上一个可回滚版本。
