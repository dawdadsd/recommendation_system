下面给你一套适合你当前项目的、比较优雅的生命周期设计。

**目标**
这 3 张表都不是永久仓库，而是推荐训练链路里的不同层次缓存：

- `user_item_preference_delta`：短期事实账本
- `user_item_preference`：训练输入快照/压缩层
- `user_cf_recall`：模型输出结果层

最佳实践不是“都保留”，而是：

**短期保留事实，中期保留训练输入，极短期保留结果版本。**

---

# 1. 三张表的定位与保留周期

## 1.1 `user_item_preference_delta`

**定位**

- 记录窗口级增量事实
- 支持幂等、追溯、重算、时间衰减

**建议保留周期**

- 当前项目推荐：**30 天**
- 更保守：15 天
- 业务量很小可放宽到 45 天，但不建议更长

**为什么**

- 这张表增长最快
- 它不是最终训练表
- 只要你已经把它汇总进 `user_item_preference`，旧 delta 的在线价值就很低

---

## 1.2 `user_item_preference`

**定位**

- ALS 训练输入表
- 用户-商品累计偏好矩阵

**建议保留策略**
不是简单“保留多少天”，而是：

- 只保留 **最近 90 天内仍有活跃行为** 的 `user-item`
- 或者每个用户只保留 **Top 300 / Top 500** 个最高偏好商品关系
- 再叠加一个最低阈值清理：
  - `preference_score <= 1`
  - 且 `interaction_count <= 1`
  - 且 `last_window_end < 30 天`

**推荐默认**

- 时间窗口：**90 天**
- 每用户保留上限：**500 个 item**
- 低价值对清理阈值：**30 天不活跃 + 低分低次数**

**为什么**

- 训练输入不能无限累加，否则会：
  - 表越来越大
  - 兴趣漂移越来越脏
  - 半年前的行为和昨天的行为权重混在一起

---

## 1.3 `user_cf_recall`

**定位**

- ALS 每次训练后的候选结果
- 给在线召回直接读取

**建议保留周期**

- 只保留 **最新 2 个模型版本**
- 最多保留 **3 个版本**
- 不按天保留，按 `model_version` 保留

**推荐默认**

- `current`
- `previous`

**为什么**

- 线上只需要当前版本
- 保留前一个版本是为了快速回滚
- 再老的版本只会占空间，没有实时价值

---

# 2. 推荐任务设计

建议拆成 4 个任务，不要混成一个大任务。

---

## 2.1 任务 A：聚合 delta 到 preference

**任务名**
`aggregate-user-item-preference`

**职责**

- 从 `user_item_preference_delta` 读取未处理区间
- 压实到 `user_item_preference`
- 更新 `recommendation_job_checkpoint`

**执行频率**

- 当前项目推荐：**每 10 分钟或每 1 小时**
- 如果流量不大：**每小时一次** 足够

**原则**

- 只做增量汇总
- 不负责删除
- 不负责 ALS 训练

---

## 2.2 任务 B：训练前重建或裁剪 preference

**任务名**
`compact-user-item-preference`

**职责**

- 删除长期不活跃、低价值的 `user-item`
- 或按 90 天窗口重建训练输入
- 可选：限制每用户最多保留 Top500

**执行频率**

- **每天凌晨一次**

**推荐优先级**

- 当前先做“清理模式”
- 后面数据量再大时升级成“重建模式”

---

## 2.3 任务 C：ALS 训练并写 recall

**任务名**
`train-als-recall`

**职责**

- 从 `user_item_preference` 读取训练矩阵
- 训练 ALS
- 生成 `model_version`
- 写入 `user_cf_recall`

**执行频率**

- 当前项目推荐：**每天 1 次**
- 如果业务变化快：**每天 2~4 次**

**原则**

- 新版本训练完成后再切换服务版本
- 不要边训练边覆盖当前线上版本

---

## 2.4 任务 D：清理旧 delta 和旧 recall

**任务名**
`cleanup-recommendation-data`

**职责**

- 删除超过保留期的 `delta`
- 删除旧 `model_version` 的 `user_cf_recall`
- 可选：清理过期 checkpoint 或历史训练日志

**执行频率**

- **每天凌晨一次**

---

# 3. 推荐执行顺序

每天推荐按这个顺序跑：

1. `aggregate-user-item-preference`
2. `compact-user-item-preference`
3. `train-als-recall`
4. `cleanup-recommendation-data`

为什么这样排：

- 先把最新 delta 压实
- 再把训练输入清理干净
- 再训练新模型
- 最后清理历史垃圾

这个顺序最稳。

# 4. 当前项目的推荐默认值

你现在这个项目，我建议直接落这组参数。

## 4.1 `user_item_preference_delta`

- 保留：**30 天**
- 聚合频率：**每小时**
- 清理频率：**每天凌晨 3 点**

## 4.2 `user_item_preference`

- 训练窗口：**最近 90 天**
- 每用户保留：**Top 500**
- 清理频率：**每天凌晨 3:10**

## 4.3 `user_cf_recall`

- 每用户保留：**Top 200**
- 保留版本：**最新 2 版**
- 清理频率：**每次训练成功后 + 每天兜底清理**

# 5. 清理任务的优雅做法

## 5.1 不要一次性大 DELETE

错误做法：

```sql
DELETE FROM user_item_preference_delta
WHERE window_end < NOW() - INTERVAL 30 DAY;
```

这会锁太多行，容易拖垮库。

**正确做法**
分批删，比如每次 5000 或 10000 行：

```sql
DELETE FROM user_item_preference_delta
WHERE window_end < ?
ORDER BY id
LIMIT 5000;
```

循环删到 0 行为止。

---

## 5.2 用“服务版本表”控制当前模型版本

不要让 `user_cf_recall` 自己猜谁是当前版本。

建议后面补一张小表：

- `recommendation_model_version`
  - `model_name`
  - `current_version`
  - `previous_version`
  - `status`
  - `updated_at`

这样清理 `user_cf_recall` 时就知道谁能删，谁不能删。

---

## 5.3 preference 表的清理不要只看时间

不能只写：

```sql
DELETE FROM user_item_preference
WHERE last_window_end < NOW() - INTERVAL 90 DAY;
```

更优雅的是组合条件：

- 长时间未活跃
- 分数低
- 交互少

例如：

```sql
DELETE FROM user_item_preference
WHERE last_window_end < NOW() - INTERVAL 90 DAY
   OR (
       last_window_end < NOW() - INTERVAL 30 DAY
       AND preference_score <= 1
       AND interaction_count <= 1
   );
```

---

## 5.4 ALS 训练最好读取一个“视图窗口”

不要直接用全表训练。建议训练 SQL 总是带窗口：

```sql
WHERE last_window_end >= NOW() - INTERVAL 90 DAY
```

这样训练边界清晰，模型结果更稳定。

# 6. 你的项目可以怎么分阶段落地

## 第一阶段：先做最小可用

先落：

- delta 保留 30 天
- preference 保留 90 天
- recall 保留 2 个版本
- 全部任务手动触发

## 第二阶段：加定时任务

加 Spring `@Scheduled`：

- 每小时聚合
- 每天凌晨清理
- 每天训练 1 次

## 第三阶段：做版本切换与回滚

补：

- 当前模型版本表
- 训练成功后再切换线上版本
- 清理时保留上一个版本

---

# 7. 一句话最佳实践

最优雅的设计不是“每张表都一直存”，而是：

**delta 做短期事实账本，preference 做有限窗口训练输入，recall 只保留少量模型版本。**
