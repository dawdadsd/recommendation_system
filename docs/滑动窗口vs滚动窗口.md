# 滑动窗口 vs 滚动窗口：差异、业务选型与落地实践

> 适用范围：Spark Structured Streaming / Flink / Kafka Streams 等流处理系统
> 目标：回答三个问题
> 1. 两类窗口本质差异是什么？
> 2. 具体业务需求该选哪一种？
> 3. 在工程上如何配置，避免误报、漏报和性能失控？

---

## 一、先给结论

- 如果业务要求是“按固定时间段做一次精确统计”，优先用滚动窗口（Tumbling Window）。
- 如果业务要求是“任意时刻都要看到最近一段时间的状态”，优先用滑动窗口（Sliding Window）。
- 滑动窗口实时性更好，但计算和状态成本更高；滚动窗口成本更低，但有边界盲区。

---

## 二、核心定义

### 2.1 滚动窗口（Tumbling Window）

- 窗口大小 `W`，步长也为 `W`。
- 窗口之间不重叠。
- 每条事件只会进入一个窗口。

示意：`W=30s`

```text
[00:00,00:30) [00:30,01:00) [01:00,01:30)
```

### 2.2 滑动窗口（Sliding Window）

- 窗口大小 `W`，步长为 `S`，且 `S < W`。
- 窗口重叠。
- 每条事件可能进入多个窗口。

示意：`W=30s, S=10s`

```text
[00:00,00:30)
[00:10,00:40)
[00:20,00:50)
[00:30,01:00)
```

## 三、差异对比（工程视角）

| 维度 | 滚动窗口 | 滑动窗口 |
|---|---|---|
| 窗口关系 | 不重叠 | 重叠 |
| 单条事件归属 | 1 个窗口 | 多个窗口 |
| 输出频率 | `W` | `S` |
| 计算成本 | 低 | 高 |
| 状态存储成本 | 低 | 高 |
| 边界盲区 | 有 | 基本无 |
| 结果形态 | 分段快照 | 连续趋势 |
| 典型目标 | 结算、报表、对账 | 风控、限流、实时排行 |

放大系数（估算）

```text
计算放大系数 A ≈ W / S

示例：
W=10min, S=1min  -> A=10
W=30s,  S=10s    -> A=3
```

解释：滑动窗口每缩小一次步长，都会显著放大计算和状态。

## 四、用业务需求反推窗口类型

### 4.1 需要“精确切片统计”的业务：用滚动窗口

业务特征：
- 强调“某个固定时间段”的准确值。
- 不能重复计数。
- 结果通常用于报表、结算、财务或离线汇总。

典型场景：

1. 计费系统
- 需求：按整点统计 API 调用次数并计费。
- 选型：`W=1h` 滚动窗口。
- 原因：每次调用只能被计费一次，不能跨多个窗口重复计费。

2. 订单小时报表
- 需求：每小时 GMV、订单数、退款数。
- 选型：`W=1h` 滚动窗口。
- 原因：报表是自然时间段口径，业务口径清晰。

3. 批量落库/离线特征快照
- 需求：每 5 分钟汇总一次写入 OLAP 或特征库。
- 选型：`W=5min` 滚动窗口。
- 原因：避免重复写入和幂等复杂度。

### 4.2 需要“任意时刻最近状态”的业务：用滑动窗口

业务特征：
- 强调“最近 N 分钟”的连续感知。
- 不能接受窗口边界导致的漏检。
- 常用于告警、风控、热度、趋势。

典型场景：

1. 限流与反爬
- 需求：任意 1 分钟内请求数超过 100 次即封禁。
- 选型：`W=1min, S=5s~10s` 滑动窗口。
- 原因：滚动窗口会在边界处漏报（59s + 1s 分散在两个桶里）。

2. 支付风控
- 需求：任意 30 分钟内交易金额累计超过阈值触发审核。
- 选型：`W=30min, S=1min` 滑动窗口。
- 原因：攻击者不会按整点作案，必须覆盖任意时间段。

3. 热门榜单
- 需求：展示最近 10 分钟热词，每 30 秒刷新。
- 选型：`W=10min, S=30s` 滑动窗口。
- 原因：保证榜单平滑更新，不是每 10 分钟突变一次。

4. 推荐系统实时兴趣
- 需求：持续计算用户最近 15 分钟行为强度，用于在线重排。
- 选型：`W=15min, S=1min` 滑动窗口。
- 原因：兴趣是连续变化的，离散快照会有“断层感”。

---

## 五、边界盲区示例（为什么风控不用滚动窗口）

需求：任意 60 秒内请求数 > 100 报警。

请求模式：`00:50 ~ 01:10` 内集中 120 次。

滚动窗口（1 分钟）统计：
- `[00:00,01:00)` 看到 60 次
- `[01:00,02:00)` 看到 60 次
- 结果：都没超过 100，漏报。

滑动窗口（`W=60s,S=10s`）统计：
- 会出现某个窗口完整覆盖这 120 次
- 结果：正确报警。

结论：只要规则是“任意时间段”，就不能只用滚动窗口。

## 六、Spark Structured Streaming 写法对比

### 6.1 滚动窗口

```java
Dataset<Row> tumbling = parsed
    .withWatermark("eventTime", "1 minute")
    .groupBy(
        window(col("eventTime"), "30 seconds"),
        col("userId"),
        col("behaviorType")
    )
    .agg(count("*").as("eventCount"));
```

说明：只传 `windowDuration`，步长默认等于窗口大小。

### 6.2 滑动窗口

```java
Dataset<Row> sliding = parsed
    .withWatermark("eventTime", "2 minutes")
    .groupBy(
        window(col("eventTime"), "1 minute", "10 seconds"),
        col("userId")
    )
    .agg(count("*").as("eventCount"));
```

说明：第三个参数是 `slideDuration`，`slideDuration < windowDuration` 即滑动窗口。

## 七、参数设计方法（实战可用）

### 7.1 四个关键参数

- `W`（window size）：看多长历史。
- `S`（slide size）：多久更新一次结果。
- `L`（watermark lateness）：接受多晚到的数据。
- `T`（trigger interval）：引擎多久触发一次微批。

### 7.2 选型步骤

1. 先定业务 SLA
- 告警最晚允许延迟多少秒？
- 可接受漏报/误报比例是多少？

2. 定窗口大小 `W`
- 取能体现业务语义的时间范围。
- 例如反爬常见 `30s/60s`，风控常见 `10min/30min`。

3. 定步长 `S`
- 一般取“可接受检测延迟”的上限。
- 例如希望 10 秒内发现异常，就令 `S<=10s`。

4. 估算成本
- `A≈W/S`，若 `A>10` 通常要谨慎，先压测再上线。

5. 定 Watermark `L`
- 用链路延迟的 P99 作为基线，再加安全余量。
- 例：P99 到达延迟 40 秒，可先设 `L=60s`。

6. 定 Trigger `T`
- `T` 不等于 `S`，但常设为 `T<=S`。
- 若 `T >> S`，窗口虽然细，但更新仍然慢。

## 八、常见误区

1. 把 Trigger 当成滑动步长
- 错误认知：`trigger(10s)` 就是滑动窗口 10 秒。
- 实际：滑动步长由 `window(..., ..., slide)` 决定。

2. 滑动窗口结果直接累加
- 同一事件在多个窗口出现，直接累加会重复计算。
- 需要明确“窗口内指标”与“全局累计指标”是两套口径。

3. Watermark 过大
- 虽然减少迟到丢弃，但会持有更多历史状态，内存压力变大。

4. 所有场景都用滑动窗口
- 滑动不是“更高级”，只是“更连续”。
- 报表、结算、对账等场景优先滚动窗口。

---

## 九、针对本项目（实时推荐）的建议

你的当前链路是用户行为流聚合，建议采用“双流策略”：

1. 主特征流（保持现状）
- 目标：稳定、低成本地产出离散特征快照。
- 选型：滚动窗口 `W=30s`。

2. 异常检测流（新增）
- 目标：及时发现任意时间段的突发刷量。
- 选型：滑动窗口 `W=60s,S=10s`。

3. 推荐实时重排流（可选）
- 目标：更新“最近兴趣”权重，提升时效性。
- 选型：滑动窗口 `W=15min,S=1min`。

这样可以在性能与实时性之间取得平衡，不需要把所有计算都改成滑动窗口。

## 十、速查表

| 你要解决的问题 | 推荐窗口 |
|---|---|
| 每小时统计并落库 | 滚动窗口 |
| 计费、对账、财务口径 | 滚动窗口 |
| 任意 1 分钟内是否超阈值 | 滑动窗口 |
| 实时风控、反爬、限流 | 滑动窗口 |
| 热榜每 30 秒刷新，但看最近 10 分钟 | 滑动窗口 |
| 推荐系统“最近兴趣”连续更新 | 滑动窗口 |

最终判断标准只有一句话：
- 关注“固定时间段统计”选滚动。
- 关注“任意时间段连续感知”选滑动。
