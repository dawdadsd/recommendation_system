### 四类召回在本项目里的实现方式

#### 1. 协同过滤召回

- 用 Spark MLlib 的 ALS，你项目依赖里已经有 `spark-mllib` 了。
  - `/C:/Users/admin/Desktop/recommendation_system/backed/pom.xml:113`
- 做法是把行为事件先汇总成 `user_item_preference(user_id, item_id, score)`，比如 view=1, click=2, cart=4, purchase=6，然后按小时或每天训练一次 ALS，输出每个用户 Top200 到 `user_cf_recall` 表或 Redis。
- 这个模块适合做“主召回”之一，但不要在接口请求时现算，必须离线/准实时产出。

#### 2. 热门商品召回

- 这个最适合继续用现在的 Kafka + Spark Structured Streaming。
- 按 1h/24h/7d 窗口统计商品热度，公式可以是：`click*1 + cart*3 + purchase*6`，再加时间衰减。结果写到 `item_hot_score` 表，在线侧放 Redis 的 ZSET。
- 首页、冷启动用户、无历史用户时，这一路非常稳。

#### 3. 相似内容召回

- 这个不要用 Spark 直接在线查。
- 更合适的是把商品 name + category + tags + description 做内容索引：
  - **MVP：** 先用 category/tags/price 规则相似，外加 MySQL FULLTEXT 或 ES 的 BM25。
  - **生产版：** 商品 Embedding + 向量检索。

> **如果你问“什么技术更好”：**
>
> - **只想少改现有栈：** 先做规则相似 + MySQL 查询。
> - **想长期效果更好：** 加 OpenSearch/Elasticsearch，因为它同时支持关键词检索、过滤、排序，后面还能走 hybrid search。
> - **如果你只关心纯向量：** Milvus/pgvector 也行，但和商城筛选条件结合时，ES/OpenSearch通常更顺手。

#### 4. 用户近期行为召回

- 这个是你现在最接近完成的一路。
- 现有 Spark 代码已经按 userId + itemId 做了近期偏好分数流，但它现在更像“把用户最近碰过的商品排个序”，还没做“扩展召回”。
- **正确做法是：**
  1.  先维护用户最近正反馈商品集合
  2.  再基于这些商品去扩展相似商品
  3.  过滤已购买、重复曝光、无库存商品
  4.  结果写 Redis，按用户在线取 TopN
- 这一路在实时场景里通常最强，尤其对“刚刚看过、加购过”的意图捕获非常准。

### 我对你这个项目的建议顺序

1.  第一步先补 `RecommendService`，不要再让 `ChatService` 直接面对全量商品。
2.  第二步先落 **近期行为召回 + 热门召回**，因为你现有 Spark 流最容易接上。
3.  第三步再补 **ALS 协同过滤召回**，让 Spark 真正发挥作用。
4.  第四步再上 **内容相似召回**，先规则版，后向量版。
5.  最后才让 **AI 做 Top50 候选** 的重排和推荐理由生成。

### 哪种技术最好

- **协同过滤：** Spark ALS 最适合，不该交给 AI。
- **热门/近期行为：** Kafka + Spark + Redis 最合适。
- **相似内容：** OpenSearch/Elasticsearch 最合适，MVP 可先用规则。
- **AI：** 最适合做“理解用户当前意图 + 重排 + 解释”，不适合单独承担召回。
